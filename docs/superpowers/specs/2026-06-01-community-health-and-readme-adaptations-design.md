# Community-health files & README adaptations — Design

**Date:** 2026-06-01
**Status:** Approved (pending written-spec review)

## Summary

Add two community-health files (`SECURITY.md`, `CODE_OF_CONDUCT.md`) and fold a few
distinctive elements from the
[microsoft/reportingservicestools](https://github.com/microsoft/reportingservicestools)
README into this project's README, adapting **structure and tone** — not Microsoft's
organisation-specific reporting channels — for this MIT-licensed community repo
(`ravenbix/IntegrationServicesTools`).

The README is autogenerated (`README.template.md` → `README.md` via
`.build/Build-SsisReadme.ps1`), so README changes split into template edits plus a
generator change (the command index moves from bullets to a table), and `README.md` is
regenerated.

## Motivation

The user likes aspects of the RS repo's README, code of conduct, and security policy. RS
is a Microsoft-owned repo: its CoC is a short pointer to
`opensource.microsoft.com/codeofconduct`, and its SECURITY.md routes to Microsoft's MSRC /
`secure@microsoft.com`. Those channels belong to Microsoft and cannot be copied verbatim
here. We adopt the *shape* (a clear vuln-reporting policy, a standard CoC, an RS-style
version table / Motivation section / tabular command list) with this project's own
channels.

## Decisions (locked)

| Topic | Decision |
| --- | --- |
| Vulnerability reporting | GitHub **private vulnerability reporting** (Security tab → *Report a vulnerability*); no public issues for security bugs. |
| Security response SLA | Best-effort acknowledgement, **no fixed SLA**. |
| Code of Conduct | **Contributor Covenant v2.1** (full text). |
| CoC enforcement contact | Routed privately through GitHub to the maintainer (the repo's private vulnerability-reporting channel — the available private inbox). |
| File location | Repo **root** (`SECURITY.md`, `CODE_OF_CONDUCT.md`). |
| README elements | Motivation section; Supported SQL Server versions table (**2012 → 2025**); footer links to CoC + Security; command index rendered as a **table**. |

## Components

### 1. `SECURITY.md` (new, repo root)

RS/MSRC structure, this project's channel. Sections:

- **Reporting a vulnerability** — use GitHub private vulnerability reporting (Security tab →
  *Report a vulnerability*). Explicit: do **not** open public issues for security bugs.
- **What to include** — vulnerability type, affected file paths, reproduction steps,
  potential impact (the MSRC checklist, reworded).
- **What to expect** — best-effort acknowledgement as soon as the maintainer is able;
  the reporter is kept informed of progress toward a fix; coordinated/responsible
  disclosure is requested. No fixed SLA.
- **Supported versions** — a short table noting that the latest published release line
  receives security fixes.
- **Setup note (doc-only):** a closing reminder that GitHub private vulnerability
  reporting must be enabled in the repository's *Settings → Code security* for the
  channel to work. Claude cannot toggle this setting; it is the maintainer's action.

### 2. `CODE_OF_CONDUCT.md` (new, repo root)

Full **Contributor Covenant v2.1** text, unmodified except for the enforcement-contact
line: violations are reported privately to the project maintainer through GitHub (via the
repo's private vulnerability-reporting channel). Retains the standard pledge, standards,
enforcement responsibilities, scope, enforcement guidelines (Correction / Warning /
Temporary Ban / Permanent Ban), and attribution footer.

### 3. `README.template.md` edits

- **Motivation** — new section after *Overview*: the SSMS click-through / hand-written
  T-SQL pain this module removes, and who benefits.
- **Supported SQL Server versions** — a table near *Requirements*:

  | SQL Server release | Database engine version | SSISDB (Project Deployment Model) |
  | --- | --- | --- |
  | 2012 | 11.0 | ✔ (SSISDB introduced) |
  | 2014 | 12.0 | ✔ |
  | 2016 | 13.0 | ✔ |
  | 2017 | 14.0 | ✔ |
  | 2019 | 15.0 | ✔ |
  | 2022 | 16.0 | ✔ |
  | 2025 | 17.0 | ✔ |

  With a note that **LocalDB cannot host SSISDB**.
- **Footer links** — add *Code of Conduct* (`CODE_OF_CONDUCT.md`) and *Security*
  (`SECURITY.md`) references in the Contributing / footer area.
- The `<!-- SSIS:COMMANDS -->` token and surrounding prose are unchanged; only the
  generated block's *format* changes (see §4).

### 4. Generator change — `.build/Build-SsisReadme.ps1` (TDD)

`ConvertTo-SsisReadme` currently emits, per group:

```
### Catalog
- **Get-SsisCatalog** — Gets the SSISDB catalog.
```

It will instead emit a markdown table per group:

```
### Catalog

| Command | Synopsis |
| --- | --- |
| **Get-SsisCatalog** | Gets the SSISDB catalog. |
```

Unchanged: the `## Command reference` heading, the `IntegrationServicesTools exposes N
commands.` line, group `### Headers`, group ordering, and verb ordering within a group.

**Test impact** (`tests/Unit/Build-SsisReadme.tests.ps1`):

- The assertion matching the bullet line (`\*\*Get-SsisFolder\*\* $dash Gets folders\.`)
  is updated to the table-row form (`\| \*\*Get-SsisFolder\*\* \| Gets folders\. \|`).
- Group-presence / ordering assertions (`### Environment`, Catalog-before-Environment,
  Get-before-New) remain valid and stay.
- TDD order: update the test to express the table format (**RED**), change
  `ConvertTo-SsisReadme` (**GREEN**), refactor if needed.

Then regenerate `README.md` (`./build.ps1 -Tasks Generate_Readme`, or
`Update-SsisReadme`) so the committed README matches the template + generator, keeping
`tests/QA/Readme.tests.ps1` (the sync check) green.

## Out of scope (YAGNI)

- `CONTRIBUTING.md` (not requested; the README already documents the build/test/commit
  workflow and points to `CLAUDE.md`).
- Issue / PR templates, `.github/` config.
- Any change to command behaviour, public surface, or the format `.ps1xml`.
- Enabling the GitHub private-advisory repo setting (maintainer action; documented in
  SECURITY.md).
